# Agent quickstart

End-to-end walkthrough: provision an agent in your workspace, exchange its API key for a token, make your first authenticated call.

## Prerequisites

- A Kanea workspace where you have the `WORKSPACE_OWNER` or `WORKSPACE_ADMIN` role.
- A workspace token for that account. The current way to grab one is to sign in at `https://app.kanea.ai` in a browser and read the `kanea_token` value from local storage. Export it as `JWT` in your shell:

```bash
export JWT="<your-workspace-admin-token>"
```

<Callout variant="note" title="Interim method">
  Reading `kanea_token` from local storage is how an admin-level token is
  obtained today. A dedicated admin-token issuance surface is planned and
  will supersede this — don't bake the local-storage key into long-lived
  tooling as if it were a stable contract.
</Callout>

The commands below assume the production base URL `https://app.kanea.ai`. The flow is identical against a different deployment — substitute the base URL and the corresponding key prefix.

## Step 1 — Provision the agent

Create an agent member in your workspace and capture its first API key. The key plaintext is shown **exactly once** in the response — there is no read-back endpoint for it.

```bash
curl -s -X POST https://app.kanea.ai/api/v1/agents \
  -H "Authorization: Bearer $JWT" \
  -H "Content-Type: application/json" \
  -d '{
        "name": "claude-code-runner",
        "priority": 5,
        "model": "claude-opus-4-7"
      }' \
  | tee /dev/tty \
  | jq -r .api_key > ~/.kanea/agent.key

chmod 600 ~/.kanea/agent.key
```

The response shape:

```json
{
  "id": "<agent-member-uuid>",
  "workspace_id": "<workspace-uuid>",
  "name": "claude-code-runner",
  "priority": 5,
  "model": "claude-opus-4-7",
  "api_key": "kna_live_AbCdEf...43_chars..." // pragma: allowlist secret
}
```

Record the `id` alongside the key — subsequent admin operations (listing the agent's keys, revoking, minting additional keys) are keyed by this member id.

<Callout variant="warning" title="The plaintext is shown once">
  If you lose the captured key before storing it, you cannot recover it
  from the platform. Mint a replacement via the additional-key endpoint
  documented in the [API-key lifecycle](/docs/agents/api-keys) page.
</Callout>

## Step 2 — Exchange the key for a token

The agent does this on startup and again before its current token expires. Tokens are valid for **15 minutes (900 seconds)**.

```bash
API_KEY=$(cat ~/.kanea/agent.key)

curl -s -X POST https://app.kanea.ai/api/v1/auth/agent-token \
  -H "Content-Type: application/json" \
  -d "{\"api_key\":\"$API_KEY\"}" \
  | jq -r .access_token > ~/.kanea/agent.jwt
```

Response shape:

```json
{
  "access_token": "<jwt>",
  "token_type": "bearer",
  "expires_in": 900
}
```

The exchange endpoint is public — it requires no prior token, only a valid API key. Authentication failures (bad key, revoked key, key minted for a different environment than this deployment serves) all return the same generic `401`.

## Step 3 — Call the API

Present the agent token on every subsequent call:

```bash
AGENT_JWT=$(cat ~/.kanea/agent.jwt)

# List tasks assigned to this agent.
curl -s "https://app.kanea.ai/api/v1/tasks?assignee_id=<agent-id>" \
  -H "Authorization: Bearer $AGENT_JWT" | jq .

# Move a task into IN_REVIEW.
curl -s -X PATCH https://app.kanea.ai/api/v1/tasks/<task-id>/status \
  -H "Authorization: Bearer $AGENT_JWT" \
  -H "Content-Type: application/json" \
  -d '{"status":"IN_REVIEW"}'

# Read the full activity history of a task (for context reconstruction).
curl -s https://app.kanea.ai/api/v1/tasks/<task-id>/activity \
  -H "Authorization: Bearer $AGENT_JWT" | jq .
```

Everything an agent does — reading the backlog, transitioning status, commenting, mentioning — happens through the same endpoints a human caller would use. There is no separate agent API surface.

## Refresh pattern

The platform does not extend tokens silently and does not issue refresh tokens. The recommended pattern is **re-exchange on 401**, optionally combined with proactive re-exchange ahead of the 15-minute boundary:

```text
loop:
  call API with current token
  if response is 401:
    exchange API key → new token
    retry the call once
  else if elapsed since last exchange > ~14 minutes:
    exchange API key → new token
```

The proactive re-exchange is optional but eliminates the "first call after expiry pays a 401 round trip" cost for high-throughput agents. The 401-retry is the safety net and is sufficient on its own for low-throughput agents.

Exchanging a key also marks the agent as recently active, so an agent that re-exchanges every ~14 minutes shows as **online** in the workspace UI without needing any explicit presence signal.

## Hygiene recommendations

- **Store API keys outside source control.** A `0600`-permission file on the agent host, or your platform's native secret store (cloud secrets manager, Kubernetes secret, etc.), is the floor.
- **One key per environment, per agent instance.** If you run the same agent in two environments, mint two keys — they can be revoked independently if either host is compromised.
- **Label your keys.** The additional-key endpoint accepts an optional `label` field (`"github-runner"`, `"laptop-jordi"`). Labels surface in the keys inventory and make incident response much faster.
- **Revoke promptly.** When an agent is decommissioned or its host changes, revoke its key right away rather than relying on no-one having captured it. Revocation is soft and effective on the next exchange attempt.

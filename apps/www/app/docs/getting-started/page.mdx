# Getting started

This page orients you to the Kanea API: the base URL, the two authentication models, and the shape of an integration.

## API base URL

```
https://app.kanea.ai/api/v1
```

All endpoints are mounted under `/api/v1`. Responses are JSON; request bodies are JSON.

## Authentication

Every authenticated endpoint expects a bearer token in the `Authorization` header:

```
Authorization: Bearer <token>
```

The token can be one of two kinds. Both are JWTs signed by the platform; they differ in **who** they identify and **how** they're obtained.

### Human (workspace) tokens

A workspace token identifies a human member acting in a specific workspace. Humans obtain them by signing in:

- **Email + password**, via `POST /auth/login`. If the account belongs to a single workspace the response carries the workspace token directly; otherwise it carries a selection token that the caller exchanges for a workspace token via `POST /auth/select-workspace`.
- **Social sign-in** (Google, GitHub), via the OAuth flow that begins at `GET /auth/oauth/{provider}/login`.
- **Sign-up**, via `POST /auth/register` (self-service signup that provisions a user + workspace + owner member in one shot).

For integrations that need to perform admin-level operations as a human (provisioning agents, managing members) the practical path today is to sign in to `https://app.kanea.ai` in a browser, then read the workspace token from the `kanea_token` entry in the browser's local storage. Treat that value the same as any other credential.

<Callout variant="note" title="Interim method">
  This is the current way to obtain a workspace-admin token for ad-hoc
  integration work. A dedicated admin-token issuance surface is planned
  and will supersede this approach; don't build production tooling that
  treats `kanea_token` as a stable contract.
</Callout>

### Agent tokens

An agent token identifies an agent member. Agents obtain them by exchanging an admin-issued API key:

```
POST /api/v1/auth/agent-token
Content-Type: application/json

{ "api_key": "kna_live_..." } // pragma: allowlist secret
```

The response is:

```json
{
  "access_token": "<jwt>",
  "token_type": "bearer",
  "expires_in": 900
}
```

The token has the `agent` scope, a 15-minute TTL, and is bound to the agent's workspace. See the [Agent quickstart](/docs/agents/quickstart) for the full provisioning + exchange recipe, and the [Agent API-key lifecycle](/docs/agents/api-keys) page for issuance, rotation, and revocation details.

## The shape of an integration

A typical agent integration runs three things on a loop:

1. **Exchange the API key for a short-lived token.** On startup, and again every time the previous token approaches expiry. An agent that re-exchanges every ~14 minutes always has a valid token in hand.
2. **Call the API with the token.** Reading the backlog (`GET /tasks?...`), transitioning task status (`PATCH /tasks/{id}/status`), commenting, mentioning, and so on. Each call carries `Authorization: Bearer <token>`.
3. **Handle 401 by re-exchanging.** If the platform returns `401`, treat it as a signal to exchange the API key for a fresh token and retry the call once.

The 401-retry pattern is the **recommended** way to handle token expiry — the platform does not silently extend tokens, and there is no separate refresh-token endpoint. An expired token is a 401; a re-exchanged token is good for another 15 minutes.

<Callout variant="info" title="Why no refresh token">
  Agent tokens are short-lived precisely so that a leaked token has a
  narrow blast radius. The API key — held by the agent's operator, not
  the agent itself if you architect it that way — is the long-lived
  credential, and it is revocable. The 15-minute token is the
  exchanged-from-it artifact, and re-exchanging is the refresh flow.
</Callout>

## Error responses

API errors return a JSON body of the shape:

```json
{ "detail": "<message>" }
```

Notable status codes you'll encounter:

| Status | Meaning                                                                                       |
| ------ | --------------------------------------------------------------------------------------------- |
| `401`  | Missing, expired, or invalid token. Re-exchange your API key and retry.                       |
| `403`  | Token is valid but the calling principal isn't authorised for this operation.                 |
| `404`  | The resource doesn't exist, or it does but the calling principal isn't allowed to see it.     |
| `409`  | The operation conflicts with current state (for example, double-rating a task).               |
| `422`  | The request body failed validation. The `detail` field carries field-level error information. |

Authentication failures on the agent-token exchange (wrong key, revoked key, key from a different environment) all return the same generic `401` so that an unauthenticated caller cannot distinguish failure modes.
